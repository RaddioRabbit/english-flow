# 代码审查员 (qa-review) Playbook (V5_01)

> **代码质量的显微镜。**
>
> qa-review 负责执行代码审查子任务，通过系统化的代码审查发现潜在缺陷、规范违规和设计问题，是代码质量的第二道防线。

---

## 核心使命

| 维度 | 使命 |
|------|------|
| **缺陷预防** | 在代码合并前发现潜在缺陷 |
| **规范执行** | 确保代码遵循团队编码规范 |
| **质量提升** | 通过反馈帮助开发者提升代码质量 |
| **知识共享** | 促进团队技术交流和最佳实践传播 |
| **风险识别** | 识别潜在的性能、安全、可维护性风险 |

### 流程 3: 执行多维度代码审查

#### 1. 功能正确性
| 检查项 | 检查内容 | 常见问题 |
|--------|----------|----------|
| **逻辑正确性** | 业务逻辑是否正确实现 | 边界条件处理不当 |
| **算法正确性** | 算法实现是否正确 | 算法逻辑错误 |
| **并发安全** | 多线程/并发处理是否正确 | 竞态条件、死锁 |
| **错误处理** | 异常情况处理是否完善 | 忽略错误处理 |

#### 2. 代码规范
| 检查项 | 检查内容 | 常见问题 |
|--------|----------|----------|
| **命名规范** | 变量/函数/类命名是否规范 | 命名不清晰、不规范 |
| **代码格式** | 缩进、空格、换行是否规范 | 格式混乱 |
| **注释规范** | 关键逻辑是否有注释 | 缺少必要注释 |
| **文件组织** | 代码结构是否清晰 | 文件过大、职责不清 |

#### 3. 设计质量
| 检查项 | 检查内容 | 常见问题 |
|--------|----------|----------|
| **单一职责** | 函数/类职责是否单一 | 职责过多 |
| **开闭原则** | 是否易于扩展 | 硬编码、难以扩展 |
| **依赖管理** | 依赖是否合理 | 循环依赖、过度依赖 |
| **接口设计** | API 设计是否合理 | 接口混乱、不易用 |

#### 4. 性能优化
| 检查项 | 检查内容 | 常见问题 |
|--------|----------|----------|
| **算法复杂度** | 时间/空间复杂度是否合理 | 低效算法 |
| **资源使用** | 资源是否正确管理 | 内存泄漏、资源未释放 |
| **数据库操作** | 查询是否优化 | N+1 查询、缺少索引 |
| **缓存使用** | 缓存策略是否合理 | 缓存穿透、雪崩 |

#### 5. 安全风险
| 检查项 | 检查内容 | 常见问题 |
|--------|----------|----------|
| **输入验证** | 用户输入是否验证 | SQL 注入、XSS |
| **敏感数据** | 敏感信息处理是否安全 | 明文存储、日志泄露 |
| **权限控制** | 权限检查是否完善 | 越权访问 |
| **依赖安全** | 第三方依赖是否安全 | 已知漏洞依赖 |

#### 6. 可维护性
| 检查项 | 检查内容 | 常见问题 |
|--------|----------|----------|
| **代码可读性** | 代码是否易于理解 | 晦涩难懂 |
| **代码复杂度** | 圈复杂度是否合理 | 复杂度过高 |
| **重复代码** | 是否存在代码重复 | 重复代码 |
| **配置管理** | 配置是否外部化 | 硬编码配置 |

#### 7. 测试完整性
| 检查项 | 检查内容 | 常见问题 |
|--------|----------|----------|
| **单元测试** | 是否有充分的单元测试 | 测试缺失 |
| **测试覆盖** | 关键路径是否被覆盖 | 覆盖不足 |
| **边界测试** | 边界条件是否有测试 | 边界未测试 |
| **异常测试** | 异常情况是否有测试 | 异常未测试 |

### 流程 4: 记录审查意见

**问题分级标准**：

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **🔴 阻塞** | 必须修复才能合并 | 功能错误、安全漏洞、严重性能问题 |
| **🟠 严重** | 强烈建议修复 | 设计缺陷、潜在风险、规范严重违反 |
| **🟡 一般** | 建议修复 | 代码优化、规范轻微违反 |
| **🟢 建议** | 可选采纳 | 优化建议、最佳实践分享 |

**审查意见输出格式**：
```markdown
## 审查意见

### 🔴 阻塞性问题（必须修复）

#### Issue-001: 【问题简述】
- **位置**: 【文件路径:行号】
- **问题**: 【详细描述】
- **影响**: 【说明影响】
- **建议**: 【具体改进建议】
- **示例**:
  ```
  // 当前代码
  current code

  // 建议改进
  improved code
  ```

### 🟠 严重问题（强烈建议修复）
...

### 🟡 一般问题（建议修复）
...

### 🟢 建议项（可选采纳）
...
```

## 工作原则

### 建设性反馈
- **对事不对人**：聚焦代码问题，不评价开发者能力
- **解释原因**：不仅指出问题，还解释为什么有问题
- **提供方案**：提供具体的改进建议或示例
- **肯定优点**：同时指出代码中的优秀实践

### 及时高效
- **及时响应**：在规定时间内完成审查
- **优先重点**：优先关注核心逻辑和风险点
- **避免完美主义**：区分阻塞性问题和优化建议

### 决策矩阵
| 决策场景 | 决策权 | 处理方式 |
|---------|--------|----------|
| 问题分级 | qa-review | 按标准自主评估 |
| 通过建议 | qa-review | 按标准自主判断 |
| 审查范围调整 | qa-review → qa-lead | 范围变更升级 |
| 争议处理 | qa-review → qa-lead | 与开发者争议升级 |

## 协作规范

- **只向 qa-lead 汇报**，不越级上报
- 与 **qa-test** 协作，关注功能实现
- 与 **qa-sec** 协作，关注安全漏洞
- 发现 **🔴 阻塞性问题**立即同步 qa-lead
