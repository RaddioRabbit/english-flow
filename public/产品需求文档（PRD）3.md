# 产品需求文档（PRD）

## 英语长难句解析一键生成 Agent 系统

------

## 产品概述

### 产品定位

英语长难句解析一键生成 Agent 系统是一款面向英语学习者、雅思备考者和英语教育内容创作者的智能化工具。该系统采用**两阶段工作流**：先进行**文本解析**生成翻译、句式分析、词汇解析和雅思备考建议；再进行**图像生成**将最终确定的文本内容转化为专业教学条漫图片集。

### 目标用户

1. **雅思备考学生**：需要系统学习长难句解析方法，提升听说读写能力
2. **英语专业学生**：需要深入理解英语语法结构和词汇用法
3. **英语教育内容创作者**：需要快速生成高质量的英语教学图文内容
4. **英语教师**：需要制作教学课件和辅助材料
5. **英语原著阅读爱好者**：希望深入理解经典文学作品中的复杂句式

### 核心价值

- **两阶段工作流**：先文本解析（可编辑确认），后图像生成（基于最终文本）
- **用户可控**：文本解析结果支持查看、编辑、重新生成，确保内容质量
- **数据流清晰**：图像生成严格使用用户最终确认的文本版本
- **专业解析**：基于 Kimi API 的深度语法分析，提供时态、语态、句式结构的专业解读
- **视觉呈现**：生成小红书风格的教学条漫，信息图表化，易于理解和传播
- **参考图支持**：支持上传参考图片，确保生成图片风格与参考图一致
- **模块化生成**：支持按需选择生成模块，灵活控制输出内容

### 解决的问题

1. 长难句理解困难，缺乏系统的解析方法
2. 手动制作教学图文耗时耗力
3. 传统英语学习资料枯燥，缺乏视觉吸引力
4. 雅思备考缺乏针对性的长难句训练材料
5. 图片生成风格难以控制，需要参考图引导
6. AI生成内容需要人工审核和修正

## 技术架构

### 技术栈

- **前端框架**：Next.js 14 (App Router)
- **后端服务**：Node.js + Supabase
- **Agent 框架**：Claude Agent SDK
- **大语言模型**：Kimi API (OpenAI 兼容接口)
- **图像生成**：Gemini 3.1 Flash Image (Nano Banana 2)
- **包管理器**：pnpm
- **部署平台**：Vercel

## 页面功能详述

### 1. 首页 / 输入页面

#### 页面功能

提供用户输入英语句子、原著信息、上传参考图片、选择生成模块的界面，支持三种生成模式：仅文本解析、仅图像生成（需已有文本）、全部生成。

#### UI布局

- **顶部导航栏**：Logo、导航链接（首页、历史记录、关于）
- **主输入区域**：
  - **[输入框1] 英语原句***：多行文本输入框，支持粘贴长难句（必填）
  - **[输入框2] 原著书名***：单行文本输入，如 "The Great Gatsby"（必填）
  - **[输入框3] 原著作者***：单行文本输入，如 "F. Scott Fitzgerald"（必填）
- **参考图上传区**（重要）：
  - 标题：📷 参考图片上传（可选，用于控制生成图片风格）
  - 5个上传卡片，每个包含：
    - **上传卡片1 - 句译对照图参考**
      - 上传按钮 / 缩略图预览
      - 文件状态指示（未上传 / 已上传）
      - 删除按钮（仅已上传时显示）
    - **上传卡片2 - 句式分析图参考**
      - 同上结构
    - **上传卡片3 - 句式总结图参考**
      - 同上结构
    - **上传卡片4 - 词汇解析图参考**
      - 同上结构
    - **上传卡片5 - 雅思备考图参考**
      - 同上结构
  - 说明文字：上传参考图片后，生成的图片将尽量匹配参考图的风格和布局
- **模块勾选区**（用于用户点“全部生成“按键的情况，如果用户没有点“全部生成“按键，以用户确定的“文本解析”的版本为准，勾选的模块就不启动）：
  - 标题：📋 选择要生成的模块
  - 复选框列表：
    - ☑️ 句译对照图（6宫格）- 依赖：prompt1-4
    - ☑️ 句式分析图（4宫格）- 依赖：句式分析文本
    - ☑️ 句式总结图（2宫格）- 依赖：句式分析文本
    - ☑️ 词汇解析图（6宫格）- 依赖：词汇解析文本
    - ☑️ 雅思备考图（4宫格）- 依赖：雅思备考建议文本
  - 全选/取消全选按钮
- **操作按钮区**（两个主按钮）：
  - **[按钮1] 文本解析**（蓝色）
    - 仅执行第一阶段：生成所有文本内容
    - 生成内容：翻译 + prompt1-4 + 句式分析 + 词汇解析 + 雅思建议
    - 完成后跳转到文本编辑页面
  - **[按钮2] 全部生成**（紫色）
    - 执行完整流程：文本解析 → 文本编辑 → 图像生成
    - 自动衔接两个阶段
- **底部说明**：
  - 生成流程说明
  - 预计耗时提示

#### 交互逻辑

- **[操作1] 输入英语句子**：
  - 系统实时检测句子长度
  - 自动去除首尾空白字符
  - 限制最大500字符
- **[操作2] 上传参考图片**：
  - 支持格式：JPG, PNG, WEBP
  - 单张最大：5MB
  - 上传后显示缩略图和文件名
  - 点击删除按钮可移除
  - **图片持久化**：上传后自动保存到 Supabase Storage
  - 刷新页面或重新登录后，已上传的图片自动恢复显示
- **[操作3] 勾选/取消模块**（用于用户点“全部生成“按键的情况，如果用户没有点“全部生成“按键，以用户确定的“文本解析”的版本为准，勾选的模块就不启动）：
  - 支持勾选是否生成句译对照图、句式分析图、句式总结图、词汇解析图、雅思备考图。
  - 至少勾选1个模块才能点击生成按钮
  - 未勾选的模块不会生成对应图片
  - 用户偏好保存到 localStorage
- **[操作4] 点击"文本解析"按钮**：
  - 验证必填字段
  - 创建任务记录到 Supabase（status: 'parsing'）
  - 调用文本解析工作流：
    1. 生成汉语翻译
    2. 智能分句（生成 prompt1-4）
    3. 生成句式分析文本
    4. 生成词汇解析文本（6个单词）
    5. 生成雅思备考建议文本
  - 保存所有文本到数据库
  - 跳转到文本编辑页面
- **[操作5] 点击"全部生成"按钮**：
  - 验证必填字段
  - 创建任务记录
  - 依次执行：文本解析 → 跳转到文本编辑 → 用户确认后 → 图像生成
- **[操作6] 点击示例句子**：
  - 自动填充：原句、书名、作者
  - 预勾选所有模块

### 2. 文本编辑页面

#### 页面功能

展示文本解析阶段生成的所有内容，支持用户查看、编辑和重新生成，确认后进入图像生成阶段。这是**第一阶段和第二阶段之间的关键桥梁**。

#### UI布局

- **顶部信息栏**：
  - 原句、书名、作者信息
  - 返回按钮、保存按钮、确认并生成图片按钮
- **分句结果区**（用于生成句译对照图）：
  - 标题：📖 分句结果（用于生成句译对照图）
  - **[输入框] prompt1**：英文原句第一部分（可编辑）
  - **[输入框] prompt2**：中文翻译第一部分（可编辑）
  - **[输入框] prompt3**：英文原句第二部分（可编辑）
  - **[输入框] prompt4**：中文翻译第二部分（可编辑）
  - 重新生成分句按钮
  - 说明：系统已根据连词（and/but/or）、分号等自然停顿处智能拆分
- **句式分析区**（用于生成句式分析图和句式总结图）：
  - 标题：📐 句式分析（用于生成句式分析图和句式总结图）
  - **[文本框] 时态分析**：可编辑
  - **[文本框] 语态识别**：可编辑
  - **[文本框] 句式分析详细内容**：可编辑
  - 重新生成按钮
- **词汇解析区**（用于生成词汇解析图）：
  - 标题：📚 词汇解析（用于生成词汇解析图）
  - 6个可编辑卡片，每个包含：
    - 单词原形 [音标]
    - 词性标签
    - 中文释义
    - 英文例句
    - 中文译文
  - 重新生成按钮
- **雅思备考区**（用于生成雅思备考图）：
  - 标题：🎯 雅思备考建议（用于生成雅思备考图）
  - **[文本框] 听力建议**：可编辑
  - **[文本框] 口语建议**：可编辑
  - **[文本框] 阅读建议**：可编辑
  - **[文本框] 写作建议**：可编辑
  - 重新生成按钮
- **模块勾选区**：
  - 标题：📋 选择要生成的模块（如果这边勾选的模块和“首页 / 输入页面”勾选的模块有冲突，以这边的勾选的模块为准）
  - 复选框列表：
    - ☑️ 句译对照图（6宫格）- 依赖：prompt1-4
    - ☑️ 句式分析图（4宫格）- 依赖：句式分析文本
    - ☑️ 句式总结图（2宫格）- 依赖：句式分析文本
    - ☑️ 词汇解析图（6宫格）- 依赖：词汇解析文本
    - ☑️ 雅思备考图（4宫格）- 依赖：雅思备考建议文本
  - 全选/取消全选按钮
- **底部操作栏**：
  - 保存修改按钮（保存为草稿）
  - 全部重新生成按钮（重新生成所有文本）
  - 确认并生成图片按钮（保存最终版本，进入图像生成）

#### 交互逻辑

- **[操作1] 编辑任意文本**：
  - 实时保存到 localStorage（草稿）
  - 显示"已修改"标记
- **[操作2] 点击重新生成按钮（单个模块）**：
  - 仅重新生成对应部分
  - 保留其他部分不变
  - 更新数据库对应字段
- **[操作3] 点击"保存修改"**：
  - 将所有文本保存到 Supabase
  - 任务状态更新为 'edited'
  - 显示保存成功提示
- **[操作4] 勾选/取消模块**（如果这边勾选的模块和“首页 / 输入页面”勾选的模块有冲突，以这边的勾选的模块为准）
  - 支持勾选是否生成句译对照图、句式分析图、句式总结图、词汇解析图、雅思备考图。
  - 至少勾选1个模块才能点击生成按钮
  - 未勾选的模块不会生成对应图片
  - 用户偏好保存到 localStorage
- **[操作5] 点击"确认并生成图片"**：
  - 保存所有文本到 Supabase（标记为最终版本）
  - 任务状态更新为 'ready_for_generation'
  - 跳转到任务详情页
  - 开始图像生成流程

### 子 Agent 详细说明

#### Agent 1: 翻译 Agent (Translation Agent)

**功能**：生成英语句子的汉语翻译

**输入**：

- 英语原句
- 原著书名
- 原著作者

**输出**：

- 汉语翻译文本（保存到 task.translation）

**提示词模板**：

```plain
你是一位精通英语文学的翻译专家。请将以下出自《{book}》的英语句子翻译成地道、准确的中文：

原句：{sentence}

要求：
1. 忠实原文含义，保持文学性
2. 符合中文表达习惯
3. 保留原文的语气和风格

请直接输出翻译结果，不要添加解释。
```

------

#### Agent 2: 分句 Agent (Segmentation Agent)

**功能**：将一个英语原句拆分为两个长短差不多的句子，并对应生成中文翻译

**输入**：

- 英语原句
- 汉语翻译

**输出**：

- prompt1: 英文原句第一部分（保存到 task.prompt1）
- prompt2: 中文翻译第一部分（保存到 task.prompt2）
- prompt3: 英文原句第二部分（保存到 task.prompt3）
- prompt4: 中文翻译第二部分（保存到 task.prompt4）

**分句规则**：

- **必须拆分**：将一个英语原句拆成长短差不多的两个英语句子
- **拆分逻辑要丝滑**：不能有一种句子拆分的突兀感
- **拆分位置**：在语气差不多有停顿的时候拆分，例如：
  - 连词后面：and, but, or, so, yet, for, nor
  - 分号（;）后面
  - 逗号 + 连接副词：however, therefore, moreover, nevertheless
  - 从句边界：after, before, when, while, although, though, because, since, if, unless 等引导的从句后
- **对应关系**：中文翻译要和拆分后的两个英文原句对应好

**提示词模板**：

```plain
你是一位专业的英语句子分析专家。请将以下英语原句拆分为两个部分，并对应拆分中文翻译。

英语原句：{english_sentence}
中文翻译：{chinese_translation}

拆分要求：
1. 必须将一个完整的英语原句拆分为两个部分
2. 两部分长度要差不多（字数差异不超过30%）
3. 拆分点要自然，在语气停顿处，如连词（and/but/or）、分号（;）、从句边界等
4. 拆分后的两部分都要有完整的意思，不能突兀
5. 中文翻译也要对应拆分成两部分，与英文对应

请按以下格式输出：

prompt1: [英文原句第一部分]
prompt2: [中文翻译第一部分]
prompt3: [英文原句第二部分]
prompt4: [中文翻译第二部分]

说明：[简要说明拆分理由，如"在and处拆分，前半句描述...，后半句描述..."]
```

#### Agent 3: 图片描述提示词 Agent (Image Prompt Agent)

**功能**：生成符合原文场景的图片生成提示词，生成的图像用于“句译对照”图片的场景描述的图像展示，放在图片的第五和第六宫格合并的一整个宫格中

**输入**：

- 英语原句
- 原著书名

**输出**：

- 英文图片描述提示词（用于 Nano Banana 生成）

**提示词模板**：

```plain

```

#### Agent 4: 语法精析 Agent (Grammar Analysis Agent)

**功能**：生成句子的详细语法分析

**输入**：

- 英语原句
- 原著书名

**输出**：

- 时态分析
- 语态识别
- 句式分析（含完整结构总结）
- 所有内容保存到 task.grammar_analysis

**提示词模板**：

```plain
#### 语法精析 

**时态分析**
全句使用**[时态名称]**（[具体动词形式]），用于描述[该时态的功能和作用]

**语态识别**
**[主动语态/被动语态]**（[具体动词示例]）

**句式分析**

- [句子成分1]：[详细说明]
- [句子成分2]：[详细说明]
- [句子成分3]：[详细说明]
- **[完整句式结构总结，必须加粗]**

【格式说明及示例】：

**时态分析格式示例**：
全句使用**一般过去时**（was, lay），用于描述过去发生的状态和存在。
或：全句使用**现在完成时**（have done），用于强调过去动作对现在的影响。

**语态识别格式示例**：
**主动语态**（the road lay / it lumbered）
或：**被动语态**（was written / were made）

**句式分析格式示例**：
- 主句：The Dover road lay beyond the Dover mail
- 插入语：as to him（介词短语作插入语，表示"就他而言"）
- 方式状语从句：as it lumbered up Shooter's Hill（由as引导，表示"当...时候"）
- **主语（The Dover road） + 谓语（lay） + 表语（beyond the Dover mail） + 插入语（as to him） + 方式状语从句（as it lumbered...）**

【关键要求】：
1. 时态名称、语态名称必须用**加粗**
2. 句式分析最后一行的完整结构总结必须用**加粗**
3. 格式必须与上述示例完全一致
```

**提示词模板**：

```plain
[解析句中的核心词汇和难点词汇，每个词汇必须包含以下信息：

- **词汇原形** [音标]

注意：本句至少解析 6 个核心词汇，重点关注《{book_name}》中的特殊用法、高级词汇、固定搭配等]

参考上下文信息：{context}
```

## 数据模型

### Task 表

```sql
create table tasks (
  id uuid default gen_random_uuid() primary key,
  user_id uuid references auth.users,
  
  -- 输入信息
  sentence text not null,           -- 英语原句
  book_name text not null,          -- 原著书名
  author text not null,             -- 原著作者
  
  -- ========== 第一阶段：文本解析结果 ==========
  
  -- 翻译结果
  translation text,                 -- 汉语翻译
  
  -- 分句结果（用于生成句译对照图）
  prompt1 text,                     -- 英文原句第一部分
  prompt2 text,                     -- 中文翻译第一部分
  prompt3 text,                     -- 英文原句第二部分
  prompt4 text,                     -- 中文翻译第二部分
  
  -- 句式分析（用于生成句式分析图和句式总结图）
  grammar_analysis text,            -- 句式分析文本（时态+语态+句式结构）
  
  -- 词汇解析（用于生成词汇解析图）
  vocabulary_analysis text,         -- 词汇解析文本（6个单词）
  
  -- 雅思备考（用于生成雅思备考图）
  ielts_tips text,                  -- 雅思备考建议文本
  
  -- ========== 配置信息 ==========
  
  -- 模块选择
  modules jsonb default '["translation", "grammar", "summary", "vocabulary", "ielts"]',
  
  -- 参考图片路径
  reference_images jsonb default '[]',
  
  -- ========== 状态管理 ==========
  
  -- 状态流转：pending → parsing → parsed → edited → generating → completed
  status text default 'pending',
  
  -- 当前阶段
  current_stage text default 'pending',
  
  -- 进度百分比
  progress integer default 0,
  
  -- 时间戳
  created_at timestamp default now(),
  updated_at timestamp default now(),
  completed_at timestamp
);

-- 状态说明：
-- pending: 待开始
-- parsing: 文本解析中
-- parsed: 文本解析完成，待用户编辑
-- edited: 用户已编辑/确认
-- generating: 图像生成中
-- completed: 全部完成
-- failed: 失败
```

### GeneratedImage 表

```sql
create table generated_images (
  id uuid default gen_random_uuid() primary key,
  task_id uuid references tasks on delete cascade,
  
  -- 图片类型，对应不同的源文本
  image_type text not null, 
  -- 'translation' -> 使用 prompt1-4 生成
  -- 'grammar' -> 使用 grammar_analysis 生成
  -- 'summary' -> 使用 grammar_analysis 生成

  storage_path text not null,
  
  -- 源文本引用（用于追溯）
  source_text text,  -- 生成该图片时使用的源文本内容快照
  
  created_at timestamp default now()
);
```

## Claude Agent SDK 能力在本项目中的具体使用方式

## Tool：工具能力

Claude Agent SDK 支持工具调用，Claude 会根据任务决定是否调用工具，而开发者决定工具是否可用、返回什么、是否允许执行。

本项目中的 Tool 分成三层：

第一类：内置工具
 用于开发与调试，主要包括读文件、执行命令、代码编辑等，不直接暴露给业务页面。

第二类：业务 Tool
 由项目实现，供主 Agent 与子 Agent 调用。建议至少定义：
 `create_task`
 `get_task`
 `update_task_step`
 `save_text_result`
 `save_image_result`
 `load_reference_images`
 `retry_task_step`
 `fetch_book_context`
 `save_prompt_version`

第三类：MCP Tool
 来自 MCP Server，例如外部网页检索、项目知识检索、数据库查询等。

本项目中最重要的原则是：**任务编排不要靠一个巨大的 orchestrator if/else 硬写，而要通过工具化的接口让主 Agent 和子 Agent 调用。**

## Skill：技能能力

Anthropic 官方 Skills 文档明确写到：
 Skill 不是通过 SDK 代码动态注册的，而是文件系统工件；若要在 SDK 中使用 Skill，必须显式设置 `settingSources` 去加载用户级或项目级的 filesystem settings；如果使用 allowlist，还必须让 `"Skill"` 出现在 `allowedTools` 中。

本项目中的 Skill 主要用于图像生成，尤其是：
 `gemini-image-generation`

目录规范必须写死为：

```
.claude/
  skills/
    gemini-image-generation/
      SKILL.md
      scripts/
        generate_image.py
```

你现有 PRD 已经把图片生成设计成统一走 Gemini Image Generation Skill，而且现有参考代码也是通过 `.claude/skills/gemini-image-generation/scripts/generate_image.py` 调用 Python 脚本来生成图片。

因此，本项目必须规定：

所有图片生成统一走 Skill Adapter。
 页面和 route handler 不允许直接调用 Python。
 图像子代理只负责生成 prompt 与调用 Skill，不负责直接实现图片逻辑。
 使用 Skill 时，Claude Agent SDK 会以 filesystem artifact 的方式发现它，而不是在业务代码里内联注册。

## MCP：外部工具接入能力

Anthropic 官方说明，Claude Code / Agent SDK 可以通过 MCP 接入外部工具、数据库和 API，MCP 是连接外部系统的标准方式。

本项目中，MCP 的作用不应是替代所有业务 CRUD，而应重点用于增强上下文：
 查询原著上下文

因此，本项目建议两类 MCP：

`web-search-mcp`

`project-data-mcp`
 用于把任务数据、模板配置、参考图索引以统一工具接口暴露给 Agent

你原 PRD 里的 Image Prompt Agent 已经明确希望“使用 claude agent sdk 的 WebSearch tool 搜索原文上下文”，这非常适合落在 MCP / 工具层。

Anthropic 官方 Subagents 文档明确说明，Subagent 用于隔离上下文、并行执行分析、应用专门指令，且 `Task` 工具已经更名为 `Agent`。

## 主工作流设计

### 模式一：文本解析工作流

输入为英语原句、书名、作者。
 先执行 Translation Subagent 输出 translation。
 再执行 Segmentation、Grammar、Vocabulary、IELTS 等子代理，输出 `prompt1-4`、`grammar_analysis`、`vocabulary_analysis`、`ielts_tips`。
 结果写入任务表与步骤表。
 这与你原 PRD 中的第一阶段文本解析流保持一致。

### 模式二：图像生成工作流

输入为任务 ID、最终确认文本、勾选模块、参考图。
 从数据库读取最终文本版本。
 分别触发对应图像子代理。
 每个图像子代理准备 prompt 后，调用 Gemini Image Skill。
 结果保存到图片表与资产表。
 这与你原 PRD 中的第二阶段图像生成流保持一致。

### 模式三：全部生成工作流

完整路径为：
 文本解析 → 文本编辑确认 → 图像生成
 这是当前产品最重要的业务路径。

## Claude Agent SDK 驱动下的工程实现规则

### 必须由 Orchestrator 统一调度

主编排必须集中在 `orchestrator.ts`，禁止页面直接调模型，禁止 route handler 里塞满所有流程。

### 每个子代理单职责单文件

子代理的 prompt、输入输出和工具权限都应单独维护，不共用大 prompt。

### Skill 只能通过 Adapter 使用

调用图片 skill 只能经过 `image-skill-adapter.ts`，不允许页面或路由层直接调 Python 脚本。

### Tool 和 Service 分离

Tool 负责面向 Agent 暴露能力；Service 负责真正操作数据库、存储和外部 API。

### Prompt 单独存放

所有文本提示词、图像提示词、系统提示词都必须独立文件存放，不能散落在 route handler 中。你现有 PRD 已经把多个关键提示词模板写清楚，这为后续提取成 `src/lib/prompts/` 提供了直接依据。

### 所有步骤都必须入库

所有步骤都必须写入任务步骤表，确保可追踪、可重跑、可审计。

```plain

**语态识别**

- [句子成分2]：[详细说明]

【格式说明及示例】：

**时态分析格式示例**：
全句使用**一般过去时**（was, lay），用于描述过去发生的状态和存在。
或：全句使用**现在完成时**（have done），用于强调过去动作对现在的影响。

**语态识别格式示例**：
**主动语态**（the road lay / it lumbered）

- **主语（The Dover road） + 谓语（lay） + 表语（beyond the Dover mail） + 插入语（as to him） + 方式状语从句（as it lumbered...）**

【关键要求】：
1. 时态名称、语态名称必须用**加粗**
2. 句式分析最后一行的完整结构总结必须用**加粗**
3. 格式必须与上述示例完全一致
```
