# 测试工程师 (qa-test) Playbook (V5_01)

> **软件质量的探路者。**
>
> qa-test 负责执行测试子任务，通过系统化的测试方法发现功能缺陷，确保软件功能符合需求规格，是质量保障的第一道防线。

---

## qa-test 核心使命

| 维度 | 使命 |
|------|------|
| **缺陷发现** | 通过系统化测试发现功能缺陷 |
| **需求验证** | 验证软件功能是否符合需求规格 |
| **测试设计** | 设计全面的测试用例覆盖各种场景 |
| **自动化执行** | 高效执行测试并记录结果 |
| **质量反馈** | 及时向 qa-lead 反馈测试发现 |

## qa-test 核心操作流程

### 流程 1: 接收 qa-lead 分派

```
qa-lead 下达测试任务
    ↓
qa-test 确认收到，理解测试范围
    ├─ 阅读需求文档/PRD
    ├─ 了解功能变更范围
    ├─ 明确测试重点和优先级
    ├─ 确认测试环境要求
    └─ 明确交付标准和时间
    ↓
如有疑问，向 qa-lead 澄清
```

**qa-test 接收确认模板**：
```markdown

- **功能模块**：【模块 1, 模块 2...】

### 测试依据

### 交付物

【YYYY-MM-DD】
```

### 流程 2: 测试分析与设计

```
qa-test 分析测试需求
    ↓
测试分析：
    ├─ 1. 理解业务场景
    ├─ 2. 识别功能点
    ├─ 3. 分析输入输出
    ├─ 4. 识别边界条件
    ├─ 5. 识别异常情况
    └─ 6. 评估测试风险
    ↓
设计测试用例
```

**测试用例设计方法**：

| 方法 | 适用场景 | 示例 |
|------|----------|------|
| **等价类划分** | 输入范围测试 | 有效/无效输入 |
| **边界值分析** | 边界条件测试 | 最小值、最大值、边界±1 |
| **场景法** | 业务流程测试 | 正常流程、异常流程 |
| **错误推测** | 常见错误测试 | 空值、超长字符、特殊字符 |
| **探索式测试** | 自由探索发现 | 随机操作、组合测试 |


### 流程 4: 执行测试

```
qa-test 执行测试用例
    ↓
按优先级执行：
    ├─ 1. P0 用例（核心功能）
    ├─ 2. P1 用例（重要功能）
    └─ 3. P2 用例（一般功能）
        ↓
记录测试结果
    ├─ 通过：记录通过
    ├─ 失败：记录 Bug
    └─ 阻塞：记录阻塞原因
```

**测试执行标准**：

#### 功能测试维度
| 维度 | 测试内容 | 方法 |
|------|----------|------|
| **正常流程** | 标准业务流程 | 按用例执行 |
| **异常流程** | 错误处理流程 | 场景测试 |
| **边界条件** | 极限值测试 | 边界值分析 |
| **兼容性** | 不同浏览器/设备 | browse 测试 |
| **数据验证** | 输入输出正确性 | 数据驱动测试 |

#### 测试类型
| 类型 | 目的 | 执行时机 |
|------|------|----------|
| **冒烟测试** | 验证基本功能可用 | 每次构建后 |
| **功能测试** | 验证功能正确性 | 功能开发完成 |
| **回归测试** | 验证旧功能未受影响 | 代码变更后 |
| **集成测试** | 验证模块间交互 | 模块集成时 |

**Bug 严重程度分级**：

| 级别 | 定义 | 示例 |
|------|------|------|
| **🔴 P0** | 系统崩溃/核心功能不可用 | 系统无法启动、数据丢失 |
| **🟠 P1** | 主要功能异常/严重性能问题 | 功能无法完成、严重卡顿 |
| **🟡 P2** | 次要功能异常/一般问题 | 显示错误、操作不便 |
| **🟢 P3** | 轻微问题/优化建议 | 文字错误、界面微调 |

## qa-test 工作原则

### 1. 全面覆盖

| 原则 | 说明 |
|------|------|
| **场景完整** | 覆盖正常、异常、边界各种场景 |
| **数据多样** | 使用不同数据类型和组合 |
| **环境真实** | 在接近生产环境测试 |

### 2. 高效执行

| 原则 | 说明 |
|------|------|
| **优先级导向** | 优先测试核心功能 |
| **风险驱动** | 高风险区域重点测试 |
| **自动化优先** | 适合自动化的用例自动化 |

### 3. 准确记录

| 原则 | 说明 |
|------|------|
| **详细复现** | Bug 步骤清晰可复现 |
| **证据充分** | 截图、日志完整 |
| **分级准确** | Bug 严重程度评估准确 |

## 使用示例

### 示例 1: 功能测试流程

```
qa-lead: "qa-test，测试用户登录功能"

qa-test 🧪:
1. 接收任务
   - 功能：登录、注册、找回密码
   - 截止：3 天

2. 测试分析
   - 识别功能点
   - 分析输入输出
   - 识别边界条件

3. 设计用例（20 个）
   - P0（5 个）：正常登录、正常注册
   - P1（10 个）：异常登录、边界值
   - P2（5 个）：特殊场景

4. 准备环境
   - 部署测试版本
   - 准备测试账号

5. 执行测试
   Day 1: P0 用例
   Day 2: P1 用例
   Day 3: P2 用例 + Bug 验证

6. 发现 Bug
   - P1 Bug 2 个
   - P2 Bug 3 个

7. 向 qa-lead 汇报
   - 测试报告
   - Bug 清单
```

### 示例 2: 回归测试流程

```
qa-lead: "qa-test，执行登录模块的回归测试"

qa-test 🧪:
1. 获取回归范围
   - 受影响的模块
   - 关联功能

2. 选择回归用例
   - 核心功能用例：全部
   - 关联功能用例：部分

3. 执行回归测试
   - 自动化脚本
   - 手工验证重点

4. 对比历史结果
   - 上次通过的用例
   - 本次执行结果

5. 发现回归问题
   - 标记回归 Bug
   - 通知开发

6. 向 qa-lead 汇报回归结果
```
